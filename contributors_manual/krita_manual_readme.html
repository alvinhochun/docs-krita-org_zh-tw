Krita 說明文件貢獻指南¶

歡迎查閱 Krita 說明文件貢獻指南！

如果有興趣協助撰寫和編輯 Krita 的說明文件，本頁會指導你如何參與。

Krita 是（自由免費的）開源軟體，實際上是社群參與的專案，由一眾志願工作者同心合力改進。這當然也需要我們來管理給新加入頁獻者的說明文件和教學，才能吸引更多的人來參與。以往我們將相關的文件發佈在多個分散的位置，而這裡的貢獻者手冊是我們嘗試將這些資訊都集中在同一個位置的成果。

這份說明文件包含了：

一份 Krita 的參考文件

這部分是讓使用者查找軟體各項功能說明的參考，例如「這個按鈕做的是甚麼」這類型的一些很單純直接的資訊。

一般通用概念指南

經過數年的經驗，我們發現對某些使用者而言，一份參考文件即使附上了一些應用例子也不能滿足他們的要求。我們的說明文件也需要為一些概念提供精簡的解釋，以及一些基本的工作流程介紹，例如是如何將影像準備成適合網絡發佈的格式。

我們也了解到，某些例如是色彩管理和圖層處理的概念，對於一般畫家與創作者而言是非常高深的學問。由於 Krita 是自由免費的軟體，因此其使用者較多會缺乏對數位藝術的專業知識。他們本身並沒有針對藝術創作而使用色彩管理技術或濾鏡圖層所需的認知。

除此之外，Krita 含有一些與其他軟體不同的系統，例如是筆刷預設系統、變形遮罩、Alpha 繼承、透視輔助尺等等。最後還有是一些連一般繪畫流程也不熟悉的使用者，他們沒有將一篇為 SAI 或 Photoshop 寫的教學文章活用到 Krita 的能力。

一些現有的文章和影片教學

Krita 團隊的其中一個優點在於我們與畫家與創作者們建立聯繫，並認可他們在做一些很棒極的東西。同樣地，我們也應該認可不同社群成員製作的教學資源，尤其是因為有些使用 Krita 來繪畫的方式是比較獨特的，而我們應鼓勵社群成員與其他人分享有關使用 Krita 的知識。

給首次貢獻者的指引¶

與以前使用過的 MediaWiki 平台相比，Sphinx 比較貼近 Krita 程式編寫的工作流程。

首先最重要的是與我們 Krita 團隊聯絡。你可以加入我們的「#krita」Matrix 聊天室。這裡有一個 Matrix 的簡介。到 https://webchat.kde.org 建立一個 Matrix 的帳號，然後加入 #krita:kde.org 聊天室。另外一件重要事是到 identity.kde.org 建立一個帳號，因為這是容許你登入 invent.kde.org 和 phabricator 這兩個我們用來組織 Krita 開發工作的平台。

Sphinx 的運作方式是基於使用 reStructuredText 標記編寫純文字檔案，然後將這些檔案處理並轉化成手冊。我們使用一個名為 Git 的版本管理系統來追踪說明文件的更動。

作出更動¶

由於我們使用的 Git 系統只允許特定人士直接提交更動，因此如有更動，你須要先將其提交作審核。

以編輯 (Edit) 模式建立合併請求 (Merge Request)¶

備註

這個方式只適用於沒有 KDE 原始碼儲存庫提交權限的成員。否則這會將更動直接提交到儲存庫內，不符合目前的工作指引。

建議沒有進階電腦技術的人使用。

不建議用作同時修改多於一個檔案。（如要同時修改多個檔案，請參見以 WebIDE 建立合併請求或以命令列建立合併請求。為每個檔案的更動各自建立一個合併請求也可。）

如果有大量的更動要提交，我們建議跟從下列步驟。

  1. 建立一個 KDE Identity。

  2. 登入 KDE_gitlab。

  3. 到 repository 並點選 Fork。

  4. 你將會被帶到自己的 fork，位置一般是 invent.kde.org/你的_KDE_登入名稱/docs-krita-org。

  5. 返回官方的儲存庫，確定你在瀏覽 Documentation > Krita.org Documentation 的專案而非你自己的 fork，否則此方式將不能按預期運作。

  1. GitLab 內置有一個修改檔案的功能。要存取這個功能，請到 Repository ‣ Files 的部分。

  2. 在頁面頂部應該會有一個標示已選取 master 的下拉選單。

  3. 找出希望修改的檔案，然後點選 Edit。

  4. 作出更動。（這個模式每次只容許修改一個檔案。）

  5. 在下方較小的文字輸入方塊「commit message」部分寫上一個簡單清晰的訊息來講解所作的更動。完成後點選 Commit changes 以提交更動並建立一個合併請求。請根據建立合併請求的準則頁面的指示填上欄位。

    使用這個方式的不便之處是不能即時知道有沒有標記錯誤。請自行用線上 Sphinx 編輯器檢查一下所作的更動（將檔案內容複製並貼上到該網頁內）。

注意

Edit 與 WebIDE 是兩個不同的功能，請確認你點選的是 Edit。

以 WebIDE 建立合併請求 (Merge Request)¶

建議懂得操作 Git 而又想同時修改多個檔案的人使用。

如果你不了解 branch（分支）的概念，不建議採用（取而代之，請見以編輯模式建立合併請求）。

  1. 依照上述指示登入 KDE_gitlab 和建立自己的 fork。

  2. 到你的 Fork（請確認網址含你的帳號名稱）。

  3. 確認你在 master 分支。

  4. 點選 WebIDE。這會將你帶到一個左邊顯示檔案列表，而右邊有一個用作顯示檔案內容的空位的頁面。

  5. 開啟你想修改的檔案並作出修改。

  6. 點選 Commit...（提交）。點兩下所有在 Unstaged changes 分類的檔案以將它們移到 Staged changes。

  7. 再次點選 Commit...——這會展開 commit message 的文字輸入方塊。在這個方塊內寫上一個簡單清晰的訊息來講解所作的更動作為 commit 訊息。

  8. 請確認設定正確無誤：你須要選擇 Create a new branch（branch 的名稱應為 [帳號名稱]/[極簡短的更動描述]。如已完成更動，請確認已勾選 Start a new merge request，否則之後將要另行建立一個 merge request。

  9. 點選 Stage & Commit。

  10. 正確地填上所有欄位：請參見建立合併請求的準則。

  11. 要手動建立一個 merge request 的話，請到 Krita Manual 官方專案儲存庫（確認網址不含你的帳號名稱），然後點選 Create a new merge request（在左方的綠色按鈕）。選取你的 fork 和先前用 WebIDE 建立的 branch。

備註

如你沒有在官方專案的提交權限，GitLab 將不會允許你儲存更動。如果你錯手在官方的專案內作出修改，你只會得到一個「An error occurred whilst committing your changes. Please try again.」的訊息。（Create a merge request也不能作用，因為這仍需要提交權限才能建立一個分支。）

在這個情況，請在你的 fork 內開啟 WebIDE 並手動將已修改的檔案內容複製過去。

以命令列建立合併請求 (Merge Request)¶

建議懂得 Git 並熟習使用命令列操作的人使用。

如果你不了解 branch（分支）的概念，不建議採用（取而代之，請見以編輯模式建立合併請求）。

  1. 依照上述指示登入 KDE_gitlab 和建立自己的 fork。

  2. 使用 git clone 來下載一份儲存庫複本。儲存庫 (repository) 頁面提供了 clone 所需的 URL，讓你可以把更動推送 (push) 到你的 fork。這樣做的好處是可以電腦上任何程式來修改檔案以及在本機建置文件以檢查有否出錯。（這個步驟只需要做一次。）

    # SSH 存取用：
git clone git@invent.kde.org:<username>/docs-krita-org.git
git remote add upstream git@invent.kde.org:documentation/docs-krita-org.git

# HTTPS 存取用：
git clone https://invent.kde.org/<username>/docs-krita-org.git
git remote add upstream https://invent.kde.org/documentation/docs-krita-org.git

  3. 每次作出修改前，請記得要 pull（拉取）最新的更動：

    git pull upstream master

  4. 請確認為你的更動建立一個分支 (branch)。自 2019 年起，任何更動都應由 master 分支出來。

    git checkout master

# 然後：
git checkout -b "<帳號名稱>/<極簡短的更動描述>"

  5. 作出更動後，請 commit（提交）並將其 push（推送）到你的 fork。請參閱 Forking on Gitlab 以了解如何使用 Git 命令列來做到這個工作流程。

    # 為你的系統安裝 python3-sphinx 套件。以 Ubuntu 為例：
sudo apt install python3-sphinx
# 建置文件（這會報告任何潛在問題，並讓你可以使用網頁瀏覽器來檢查更動）
make html
# 確認情況良好
git status
git diff
# 追蹤所有已變更的檔案
git add .
# 提交更動
git commit
# 把更動推送到 fork
git push

  6. 最後，到官方專案儲存庫的頁面，然後到 Merge Requests 部分。選取你的 fork 以及正確的 branch 來建立一個 merge request。請參見建立合併請求的準則內的指示以填妥欄位。

建立合併請求的準則¶

  1. Commit 訊息應符合這裡提及的標準：How to Write a Git Commit Message（如何撰寫 Commit 訊息）

  2. Title（標題）和 Description（描述）應解釋所作的更動和作出更動的原因，與 Commit 訊息相似，因此也應依照相同的標準。

  3. Target 應指向 master。

  4. 如認定該合併請求將需要作進一步修改，請以 [WIP] 作為標題開首。

  5. 請確認已勾選 Allow commits from members who can merge to the target branch.。合併請求因技術原因而經常要被 rebase（衍合）到 master 分支之上，此舉會修改合併請求，但不會改變更動的實際內容。Rebase 操作可以是由你或是由審核者來進行。如果你不想經常被要求進行 rebase 操作，請啟用此選項以讓審核者代為進行 rebase。

  6. 在多數情況下，勾選 Delete source branch when merge request is accepted 是安全的。

  7. 除非審核者提出其他要求，否則應勾選 Squash commits when merge request is accepted。Commit 訊息的第一句會取自合併請求的 Title（標題）而其餘內容會取自合併請求的 Description（描述）。

  8. 當完成建立合併請求，請到 IRC 聊天室請求有權限的成員代為設定 Needs Review 標籤。

  9. 如合併請求帶有錯處，你可能會收到回饋意見。你只需依照以下其中一個方法修正問題：

    • 如果想使用 Edit（編輯）模式，可到 Changes（更動）部份並點選鉛筆圖示（工具提示標籤為「Edit」）。

    • 如果想使用 WebIDE 模式，可到你的 Fork 選擇更動所在的分支並從該分支開啟 WebIDE。

    • 如果你在本機修改檔並使用命令列作業，請確認處於正確的分支，然後推送更動。GitLab 會自動更新合併請求。

    作出更動後，請務必請求成員再次設定 Needs Review 標籤。

若要閱讀更詳細的資訊，請參見技術分類的 Forking on Gitlab。

備註

在撰寫本文時為止，只有已獲得推送寫入權限的貢獻者才能設定合併請求的標籤。（如你不清楚這是甚麼意思，你多數是沒有這個權限。）因為這樣，每當建立或更新合併請求時你都要到 IRC（見 Krita 的社群）請求其他成員代為設定標籤。

以命令列建置說明文件¶

如果你希望在建立合併請求前首先測試一下更動的效果（而又了解如何使用 Git 和命令列），你可以依照以命令列建立合併請求中第 5 步所列出的步驟。

一般思維¶

這個章節是為了確定合適的寫作風格。寫作風格——無論是實際上還是美觀上的考慮——通常都是以一個目的或一般思維為基礎。我們希望透過這份手冊達到甚麼目的呢？這份手冊又是為誰人寫的呢？

目標讀者¶

在討論目標讀者的時候，我們不能作「所有 Krita 使用者都是五十五歲的老人」這種陳述。Krita 的使用者數目眾多，不論各行各業、男女老幼、來自不同背景的人士也有。

儘管如此，我們對使用者群體有以下的理解：

  • 他們是創作者。我們明確地以這類使用者為為目標。

    • 因為這點，我們知道他們喜歡漂亮的圖片。

    • 他們以視覺為主。

    • 他們希望製作出美觀的影像。

因此，每頁的目標就是給出可用作產生美觀的影像的功能。

除此之外，我們也察覺到下列類別的使用者：

  • 試用以不同軟體繪畫插圖的高中及大專生：他們通常有一些使用其他繪畫軟體的經驗（例如是 Paint Tool SAI 或 Photoshop），不過需要透過某些途徑去認識 Krita 的特色。這類使用者的特點是，他們經常交流並互相分享大量資訊，例如各種秘決、小技巧和教學。

  • 以數位繪畫軟體謀生的專業人士：這類使用者的特點是，他們有著豐富的認識，並樂於捐款以支持改進軟體。他們又可細分為兩類：

    • 非技術性專業人士：這類使用者對軟體的內部技術、含數學性質的部分並不太了解，但透過多年使用經驗建立了實際可靠的工作流程，並能依照直覺善用軟體。他們通常是畫家和製作印刷品的人。

    • 專業技術人士：這類使用者以 Krita 作為技術流程的一部分，關心色彩及個別像素的準確數學運算方式。他們通常在遊戲及 VFX（視覺特效）業界工作，不過間中也會有科學研究員。

  • 成年或老年的繪畫愛好者：這類使用者多數不太懂電腦技術，而且容易被教學文章省略掉的一個小步驟卡住。作為一個群體，他們的強項在於能創造出一些非比尋常的工作流程——一些學生不懂得而專業人士沒有空餘時間研究的流程——而且會以這些流程創作出很酷的作品。他們也對社群中較年輕的學生成員有著緩和作用。

在這四類使用者當中……

  • 只有一類擁有技術知識。因此我們需要提供「通用概念」這部分，為其他類別的讀者們打好基礎，並使我們可以基於這些內容之上撰寫說明文件的其他內容。

  • 有三類使用者均可能已有使用其他軟體的經驗，因此也許需要參考一些過渡指南，以協助他們適應 Krita 與其他軟體的不同之處。

  • 有兩類使用者需要知道如何配合其他軟體一同使用 Krita。

  • 有兩類使用者不理解他們在做甚麼，或許需要由他人引導才能完成最基本的步驟。

由此我們可以得出以下幾項指引：

一般書寫風格¶

盡可能使用美式英語。

我們在這份說明文件中使用美式英語，以配合 Krita 使用者介面預設使用的美式英語。

保持語句斯文有禮，但不要使用學術性語文。

作為一個社群，我們希望使用者們感到受歡迎，因此我們要避免採用不友善的用語。KDE 也不縱容使用粗話。然而，偏向另一個極端採用過於學術性的用語，會使文章內容顯得過於複雜而嚇怕使用者，因此我們也要避免這樣。

避免使用動態 GIF（具爭議性）

避免動畫的理由是，癲癇症患者可能會受快速移動變化的影像影響而病發。另外，太偏重於使用動畫來作解釋的話會容易忽略文字解釋。如果你必定要使用動態 GIF，請至少在頁面較前的介紹部分知會讀者。

使內容易於翻譯

這包括使用 SVG 製作說明圖，並為文字作出適當的標記。

有關相片與傳統畫作¶

  • 我們不鼓勵在說明文件中加入相片和傳統畫作，除非必定要用它們來解釋一個特定概念。理由是，使用 Krita 製作的現代數位藝術作品多的是。若我們在 Krita 的介面中展示林布蘭大師的名作，這反而顯得不自然，同時亦帶有一點誤導成份。Pepper&Carrot 漫畫作品全集均以 Krita 製作，而且提供了原始檔下載，因此當你沒有合適的影像可用時，可在這套漫畫中搜尋。避免加入相片，因為 Krita 的重心是繪畫軟體而不是相片編輯軟體，過度使用相片或會令讀者產生 Krita 是用來編輯相片的錯覺。

  • 當然，我們有時也希望講解一些在相片和名畫中出現的概念，例如光澤或者間接照明 。在這些情況下，請為該影像加上說明文字。如果是名畫，請在當中提及作品的名稱或其作者。如果是相片，請清楚標明。

  • 文章明顯地在講解相片接合插畫 (Photobashing) 的情境下可以使用相片。

有關使用影像的一般指引¶

  • 避免在影像中嵌入文字。取而代之，使用 figure 說明文字標記是較好的做法。

  • 如必須加入文字，請使用 SVG 影像格式以方便將來可以簡易地修改文字，或盡量減少文字量。

  • 盡可能製作高品質或美觀可愛的影像。始終 Krita 是一個用來繪畫的軟體，我們應該要給讀者一個好印象！

  • 請謹記此說明文件的授權條款為 GDPL 1.3，因此所有提交到這份文件的影像均會以其授權。如影像為創用 CC-By-Sa 或 CC-By 授權，請確認將該影像的作者和來源適當地以 figure 說明文字作標注。請勿提交不能符合這些授權條款的影像。

程序¶

在這一節我們會列出一系列工作流程。

建立標籤 (Tag) 與分支 (Branch)¶

Adding and removing text will be done in the draft branch.

Proofreading results for old pages will be considered as bugfixes and thus will go into the master branch and merged into the draft branch as necessary.

Before the draft branch is merged for a given release:

  • The master branch will be tagged with the old version.

  • The draft branch is first double checked that it has updated version number and updated epub cover.

The draft branch will not be merged until the day before a release to keep the pages intact for long enough.

Each release will have a version of the epub uploaded as part of the release process. .. Where do we get the POT files from? Even the translated versions?

移除頁面¶

如有功能在某個版本中被移除，相關頁面會：

  1. 先標記為「已棄用」。

    可使用以下方式標記：

    .. deprecated:: 版本編號

    這裡寫上使用者應該如何代替該項已放棄使用的功能。

  2. Will be linked on a page called 'deprecated'

  3. If the next version rolls around, all the pages linked in the deprecated section will be removed.

加入頁面¶

  1. 確認頁面已放到正確的位置。

  2. 依照 Krita 說明文件標記格式使用慣例以確認頁面的格式正確。

  3. 將頁面加入到 TOC（目錄）。

  4. 如該功能是在近期版本新增的功能，請加入 versionadded 標記：

    .. versionadded:: 版本編號

    有需要時選填說明文字。

與影像相同，請勿加入未獲授權使用的文字。加入到此文件的文字必須由你自行撰寫，或者你已從原作者獲得授權可在此文件中使用。此文件以 GDPL 1.3+ 授權，因此任何加入到此文件的文字均會重新以該授權發佈。

更改頁面¶

如你將頁面重寫，而非單純作校閱修訂，重寫後的頁面應先通過審核。

如因功能改動而修改頁面，而你又擁有提交 (commit) 權限，則可以跳過審核而直接將更動推送到主儲存庫（除非你認為經過審核後會較為安心）。你應在變更的部份加上：

.. versionchanged:: 版本編號

    在此處描述變更內容。

在作出任何變更後，請考慮要否在頁頂 metadata（中繼資料）內的 author（作者）欄位加入自己的作者資料。

連同版本編號加上 deprecated、versionadded 和 versionchanged 標記可讓我們能輕易地使用 grep 在說明文件中找出這些內容：

grep -d recurse versionadded * --exclude-dir={_build,locale}

發現問題頁面¶

如發現有錯處，可：

  • 按作出更動一節建立一個合併請求。

  • 在 Manual Project Workboard 看板建立一個 Task。

  • 在 bugzilla 上「Krita」專案中的「documentation」組件下建立一個 bug report。

校對¶

我們的校對工作分為兩類。

最重要的一類為審閱其他人作出的更動。在 KDE_gitlab 上有兩種方法進行審閱：

  1. 審閱合併請求

    你可以協助審閱合併請求。審閱程序一般是程式設計員互相在大家的程式碼中尋找錯處。由於說明文件和程式碼一樣都是以文字為基礎，因此我們也可以採用相同的程序來檢查錯處！

    A merge request, is an amount of changes done in a document (added, removed) put into a machine readable file. When someone submits a review request (on system like gitlab or github this is a merge or pull request), people who maintain the original files will have to look them over and can make comments about things needing to change. This allows them to comment on things like typos, over-complicated writing but also things that are incorrect. After a patch has been accepted it can be pushed into the version control system.

  2. 就文件的更動留下評語。

    Commenting changes happens after the fact. You can comment on a change by going to the commit message (from the repository page, go to history and then click on an entry), where you will be able to make comments on the changes made.

In both cases, the interface consists of the difference being shown, with on the left the old version, and on the right the new version. Lines that have been added will be marked in green while lines that have been removed will be marked with red. You can click a speech bubble icon to add an 'inline' comment.

The second major way the manual needs to be proofread is over the whole file. Many of the pages have only been checked for correctness but not for style and grammar.

For this you will need to follow the 作出更動 section, so that you can have full access to the pages and edit them.

翻譯¶

Krita 說明文件的翻譯工作是由KDE 在地化團隊負責。如果你希望參與正體中文版 Krita 說明文件的翻譯工作，請到臺灣自由暨開源軟體在地化社群的 Telegram 群組查詢。

在地化團隊成員會存取說明文件的 PO 編目檔。這是一種可以使用翻譯軟體（如 POEdit 和 Lokalize）開啟的檔案格式。翻譯者會合作翻譯這些編目檔，並將它們提交到放置翻譯檔的 SVN 儲存庫。一個每日運行一次的自動程式會將翻譯檔由 SVN 套用到說明文件內。正體中文在地化團隊目前的工作流程使用了私人的 Weblate 平台來協助翻譯。

如在地化團隊希望翻譯影像，這也是可以的。在 image 資料夾內的影像預設對應英文 (en)。要翻譯影像的話，首先在影像所在的資料夾建立一個以語言代碼命名的子資料夾（正體中文為 "zh_TW"），然後將已翻譯的影像放到裡面。例如 Sphinx 會從 /images/zh_TW/Pixels-brushstroke.png 找出 /images/Pixels-brushstroke.png 的中文版本，或從 /images/dockers/zh_TW/Krita-tutorial2-I.1-2.png 找出 /images/dockers/Krita-tutorial2-I.1-2.png 的中文版本。

已完成的翻譯語言要手動加入到建置指令稿才會在官方網站上展示。在地化團隊如對翻譯感到滿意，應透過 kimageshop 郵遞清單 (kimageshop@kde.org) 或 foundation@krita.org 與 Krita 團隊聯絡，以處理相關事宜。
