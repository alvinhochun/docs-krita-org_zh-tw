Krita 的社群¶

Krita 專案除了位於 https://www.krita.org 的官方網站之外，還有以下三個溝通途徑：

  • Internet Relay Chat (IRC) 網絡聊天窒

  • 郵遞清單 (mailing list)

  • Phabricator 平台

即使有些 Krita 開發者和使用者有使用各種網上社交平台，它們並不是我們用來討論新功能、軟體錯誤、開發工作或未來計劃的地方。

除此之外，還有：

  • 缺陷回報及追縱平台

  • 開發者衝刺聚會 (sprint)

  • Dmitry (irc: dmitryK|log)：主導開發者。全職為 Krita 工作。

Krita 團隊成員遍佈多國，不過大多數開發工作都發生於歐洲及俄羅斯時區的日間時間。

Krita 是 KDE 社群的一部分。KDE® 社群是一個致力創造開放而且易用的電腦操作體驗，為使用者提供一個先進的圖形介面，各式各樣的通訊、工作、教育和娛樂應用軟體，以及給開發者一個可以作為基礎簡易地建造新應用軟體的平台。KDE 的貢獻者指南也適用於對 Krita 的貢獻。這份指南可在這裡找到。

Krita 基金會成立的目的是支持 Krita 的發展。Krita 基金會從 2013 年就開始資助 Dmitry 為 Krita 的開發工作。

IRC 聊天室¶

IRC 是開發者們的主要溝通途徑。你可以為各個作業系統找到 IRC 用戶端。

  • 如何連線到 IRC 聊天室：連接到 Libera.Chat，選用一個暱稱然後再加入「#krita」和「##krita-chat」的聊天室即可。「#krita」是用來討論公事，而「##krita-chat」為離題討論。

  • 不要問可否發問。如果有問題的話，直接發問就可以。

  • 同時有多個討論串是很平常的事，不要怕。

  • 輸入對方的暱稱再加上冒號就可以明確地跟對方交談。

  • 每逢星期一的 16:00 CET 或 CEST（歐洲中部時間），我們會在這個聊天室舉行一個會議，討論過去一週所做的事、將來會做的事，以及任何與 Krita 專案有關的事項。會議記錄會保存到 Google 文件。

  • 聊天室最繁忙的時段是 CET 或 CEST 的日間至睡前，大約是台灣時間中午到凌晨時份。其他時段通常都是相對冷清的。

  • IRC 沒有聊天記錄，當你關閉聊天室的一刻你便會離開，而且不能再接收當中任何訊息，直到你重新加入聊天室為止。在提問之後，請務必保持聊天室開啟等待答覆。

  • 不停重複斷線並重新連線會打亂正在進行中的對話，對其他使用者造成困擾，所以請不要這樣做。

郵遞清單¶

郵遞清單通常用作發出公告，少數情況下亦會用作討論事項。任何人希望參與 Krita 的工作都應該訂閱這個郵遞清單。

郵遞清單存檔

我們的郵遞清單叫「kimageshop」，因為這是 Krita 專案最初的名稱。後來因為法律問題更名為「Krayon」然後再次更名才到「Krita」。

GitLab 平台 (KDE Invent)¶

GitLab 平台對於 Krita 團隊有以下用途：

  • 檢查貢獻者提交的合併請求（Merge Request，亦簡稱「MR」）：Graphics/Krita 是程式原始碼專案，而 Documentation/Krita.org Documentation Website 是 Krita 說明文件的內容。

  • 寄存原始碼的 Git 儲存庫：https://invent.kde.org/graphics/krita.git。留意雖然外部 GitHub 或 Phabricator 有 Krita 原始碼的專案鏡像，我們一般不會使用它們。

  • 寄存 Krita 說明文件內容的 Git 儲存庫：https://invent.kde.org/documentation/docs-krita-org

請勿在 GitLab 建立 Issue 或缺陷報告。

請把任何程式碼貢獻都以 Merge Request 方式提交到 GitLab。請勿把修補檔附加到 Bugzilla 的報告內。

Phabricator 平台¶

Phabricator 平台對於 Krita 團隊有以下用途：

  • 追縱進行中的工作：https://phabricator.kde.org/maniphest/ 這包括開發工作、新功能設計和使用者體驗設計，以及與網站相關的工作。

請勿回報軟體缺陷到 Phabricator 上。Phabricator 是我們用來計劃工作的平台。

Bugzilla 缺陷回報及追縱平台¶

Krita 與 KDE 社群的其他專案共用同一個缺陷追縱平台。Krita 相關的缺陷報告都放到「Krita」的 Product 分類之下。我們有兩種報告，分別是一般的缺陷報告 (bugs) 和功能願望請求 (wishes)。請參見回報缺陷和缺陷報告分類這兩個部分。除非你收到開發者的要求，否則請不要在 Bugzilla 建立功能願望請求。請參見功能請求的部分了解如何建立良好的功能請求。

開發者聚會 (Sprint)¶

有時，核心開發者和使用者會聚集在同一個地方——通常在荷蘭代芬特爾 (Deventer)——面對面一同做一些包括程式設計、UX 設計、網站設計等等的工作。參與開發者聚會的交通費用通常會由 KDE e.V. 資助，而住宿會由 Krita 基金會提供。
